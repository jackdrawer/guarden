  <name>Task 1: Create typed error hierarchy with user messages</name>
  <files>lib/errors/app_errors.dart</files>
  <action>
Create comprehensive error type system in `lib/errors/app_errors.dart`:

**Base error class:**
- `AppError` abstract base class implementing `Exception`
- Fields: `message` (technical), `userMessage` (friendly), `canRetry` (bool), `action` (optional recovery action string)
- toString() returns technical message for logging

**Concrete error types:**
1. `StorageError` - Keychain/Keystore failures (migrate existing class from secure_storage_service.dart)
   - Example user message: "Couldn't access secure storage. Please re-enter your master password."
   - canRetry: true, action: "re-enter password"

2. `CryptoError` - Encryption/decryption failures
   - Example: "Couldn't encrypt data. Your information is safe but wasn't saved."
   - canRetry: false, action: null

3. `NetworkError` - API call failures (RevenueCat, Pwned API)
   - Example: "Couldn't connect to server. Check your internet and try again."
   - canRetry: true, action: "retry"

4. `DatabaseError` - Hive operations failures
   - Example: "Couldn't save your data. Please check available storage."
   - canRetry: true, action: "retry"

5. `BiometricError` - Face ID/Touch ID failures
   - Example: "Biometric authentication failed. Try again or use master password."
   - canRetry: true, action: "retry"

6. `ValidationError` - User input validation failures
   - Example: "Password must be at least 8 characters long."
   - canRetry: false, action: null

**Constructor pattern:** Each error type should accept both technical and user-friendly messages. If userMessage not provided, use a sensible default.

**Why typed errors:** Allows catching specific error types and applying appropriate recovery strategies (retry for network, re-auth for storage, show validation message for input).

<task type="auto">

  <files>lib/widgets/error_handler.dart</files>
  <action>
Create centralized error handling utilities in `lib/widgets/error_handler.dart`:

**ErrorHandler class** (static utility methods):

1. `static void handleError(BuildContext context, Exception error, {VoidCallback? onRetry})`
   - Converts any Exception to AppError (or wraps in generic AppError)
   - Calls showErrorSnackBar with appropriate error
   - Wires onRetry callback if error.canRetry is true
   - Logs technical message to console (development only)

2. `static Future<T?> withErrorHandling<T>(BuildContext context, Future<T> Function() operation, {VoidCallback? onRetry})`
   - Wraps async operations in try-catch
   - Catches all errors, converts to AppError, displays via showErrorSnackBar
   - Returns null on error, result on success
   - Usage: `await ErrorHandler.withErrorHandling(context, () => service.saveData())`

3. `static AppError convertToAppError(Exception exception)`
   - Converts common exceptions to typed AppError
   - SocketException → NetworkError
   - FormatException → ValidationError
   - HiveError → DatabaseError
   - PlatformException → BiometricError or StorageError (based on exception.code)
   - Default: Generic AppError with "Something went wrong. Please try again."

**Why centralized utilities:** Ensures consistent error handling across all screens. Reduces boilerplate in UI code. Easy to add logging, analytics, or monitoring hooks later.
