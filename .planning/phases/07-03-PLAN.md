# Plan 07-03: Provider Error Handling & UI Integration

## Goal
Integrate the specific, strongly typed AppErrors (from Plan 07-01) and gracefully propagated errors (from Plan 07-02) into the Riverpod layer. We will implement global error display mechanisms (using `ErrorHandler` and `ErrorSnackBar`) to catch these errors in the presentation layer without having to rewrite the entire UI widget tree.

## Scope of Work

1. **Global Error Notification System:**
   - Instead of breaking the UI by migrating all synchronous `Notifier` providers (like `bankAccountProvider`) to `AsyncNotifier` (which would require a massive immediate UI refactoring), we will preserve the current synchronous state properties (since local Hive access is instantaneous).
   - Inject a GlobalKey or utilize a dedicated Error/Notification provider combined with a `ScaffoldMessenger` wrapper so providers can dispatch errors to the UI from anywhere.

2. **Provider Modifications:**
   - **Auth Provider (`auth_provider.dart`):** Wrap `login`, `setupMasterPassword`, and `unlockWithBiometrics`. Dispatch errors (e.g., `CryptoError`, `BiometricError`) to the UI.
   - **Premium Provider (`premium_provider.dart`):** Wrap `checkPremiumStatus` and `purchasePackage`. Dispatch `NetworkError` to the UI natively.
   - **Vault Providers (`bank_account_provider`, `subscription_provider`, `web_password_provider`):** Wrap `add`, `update`, and `delete` operations in `try-catch` blocks. If `Hive` operations fail, catch the exception, throw/dispatch a `DatabaseError`, and prevent state corruption.
   - **Settings Provider (`settings_provider.dart`):** Wrap setter methods in try-catch.

3. **Global Scaffold Messenger setup:**
   - Define a `GlobalKey<ScaffoldMessengerState>` in `main.dart` or `app.dart`.
   - Update `ErrorHandler.showError` to use this global key so errors can be shown unconditionally, independently of local BuildContext contexts inside async provider unawaited methods.

## Deferred (Phase 8/9):
- Massive `AsyncValue.guard` UI refactoring. Keeping the Local DB synchronous logic allows snappy offline behavior while still capturing and displaying errors seamlessly.
